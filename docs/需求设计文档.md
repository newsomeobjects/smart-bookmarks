## 1. 项目概述

### 1.1 项目背景

现代浏览器(Chrome、Edge等)的书签管理功能存在以下痛点:
- 新建书签默认存入"其他书签"文件夹,缺乏有效分类
- 书签数量增长后,手动整理效率低下
- 常用书签难以快速访问
- 缺乏智能化的书签管理工具

### 1.2 项目目标

开发一款基于AI的智能书签管理浏览器扩展,实现:
- **v1.0核心目标**: 实现书签批量分类和单个书签自动分类功能
- **v2.0扩展目标**: 实现书签热度排序和智能推荐功能

### 1.3 目标用户

- 书签数量较多、需要高效管理的用户
- 希望自动化整理书签的用户
- 需要快速访问常用书签的用户
- 对AI辅助工具有接受度的技术用户

## 2. 功能需求

### 2.1 V1.0版本核心功能 (高优先级)

#### 2.1.1 书签批量分类功能 ✅ 已实现

| 需求项 | 描述 |
|--------|------|
| **功能描述** | 用户主动触发,AI对书签栏中的所有书签进行智能分类整理 |
| **触发条件** | 用户在管理页面点击"开始分类"按钮 |
| **前置条件** | 1. 用户已配置AI模型参数(API密钥、端点等)<br>2. 书签栏中存在待分类书签 |
| **输入要求** | - 书签数据: 书签栏中所有书签的标题和URL<br>- 分类配置: 最大文件夹数(默认10)、最大层级深度(默认2)、批次大小(默认30) |
| **处理流程** | 1. 提示用户导出备份书签(可选)<br>2. 用户确认后继续<br>3. 获取书签栏所有书签数据<br>4. 第一阶段: AI生成分类目录树结构<br>5. 第二阶段: AI为每个书签分配分类路径<br>6. 校验分类结果完整性和有效性<br>7. 创建分类文件夹结构<br>8. 移动书签到对应分类文件夹<br>9. 发送完成通知 |
| **输出结果** | - 分类文件夹结构: 按AI生成的分类树创建的文件夹<br>- 书签分布: 每个书签被移动到对应的分类文件夹<br>- 分类报告: 包含分类统计、用户画像、校验结果 |
| **异常处理** | - API调用失败: 显示错误信息,支持重试<br>- 分类结果校验失败: 记录警告日志,继续执行<br>- 书签移动失败: 记录错误日志,跳过该书签 |
| **实现状态** | ✅ 已完整实现,包括进度反馈、错误处理、通知机制 |

### 2.2 V2.0版本核心功能 (中优先级)

#### 2.2.1 书签热度排序算法 🔮 规划中

| 需求项 | 描述 |
|--------|------|
| **功能描述** | 追踪用户访问行为,计算书签热度分数,实现高频访问书签自动前置 |
| **触发机制** | 后台持续监控用户标签页访问行为 |
| **输入要求** | - 访问事件: 标签页激活、URL变更事件<br>- 时间戳: 访问发生的时间<br>- 书签匹配: 访问URL与书签URL的匹配 |
| **热度算法** | 综合计算访问频率和最近访问时间:<br>- 访问频率权重: 60%<br>- 最近访问权重: 40%<br>- 时间衰减因子: 0.95<br>- 计算公式: `热度分数 = (访问次数/天数) * 0.6 + e^(-距今天数/7) * 0.4` |
| **输出结果** | - 热度分数: 每个书签的热度评分<br>- 排序列表: 按热度排序的书签列表<br>- 访问统计: 访问次数、首次访问、最近访问时间 |
| **数据存储** | - visitData: {bookmarkId: {visitCount, lastVisit, firstVisit, heatScore}}<br>- 追踪周期: 7天(可配置) |
| **实现状态** | 🔮 技术预研阶段,需要设计数据结构和算法 |

#### 2.2.2 书签栏智能生成机制 🔮 规划中

| 需求项 | 描述 |
|--------|------|
| **功能描述** | 自动将高热度书签复制到书签栏,方便快速访问,与正常书签区分管理 |
| **触发机制** | 定期同步(每小时)或热度变化时触发 |
| **输入要求** | - 热度排行: 基于热度算法计算的TOP N书签<br>- 书签栏配置: 最大推荐数量(默认6个,可配置) |
| **处理流程** | 1. 计算所有书签的热度分数<br>2. 选取TOP N高热度书签<br>3. 将高热度书签复制到书签栏(非移动)<br>4. 为复制的书签设置同步状态标识(syncStatus=false)<br>5. 定期更新推荐列表 |
| **输出结果** | - 热度书签: 书签栏中显示的TOP N高热度书签副本<br>- 同步标识: 通过syncStatus=false标识为热度书签,与正常书签区分 |
| **特殊规则** | - 批量整理书签时,不处理热度书签(syncStatus=false的书签)<br>- 热度书签仅作为快捷访问入口,不参与分类整理<br>- 原始书签位置保持不变,热度书签为副本 |
| **异常处理** | - 无访问数据: 不生成热度书签<br>- 书签已删除: 自动从热度书签中移除<br>- 原书签移动: 热度书签自动更新位置 |
| **实现状态** | 🔮 功能规划阶段,需要设计同步标识和过滤机制 |

## 3. 非功能需求

### 3.1 性能需求

| 需求项 | 指标 | 说明 |
|--------|------|------|
| 新建书签分类响应时间 | ≤ 5秒 | 从书签创建到分类完成的时间 |
| 批量分类进度反馈 | 实时 | 每个阶段实时显示处理进度 |
| 大量书签处理 | 分批处理 | 支持分批次处理,避免阻塞UI |
| 缓存刷新 | ≤ 2秒 | 书签缓存刷新时间 |
| 内存占用 | ≤ 50MB | 扩展运行时内存占用上限 |

### 3.2 安全需求

| 需求项 | 描述 |
|--------|------|
| API密钥存储 | 使用chrome.storage.local存储,不明文显示 |
| 数据传输 | 所有API调用使用HTTPS加密传输 |
| 用户隐私 | 书签数据仅在用户主动请求时发送给AI |
| 权限最小化 | 仅申请必要的浏览器权限 |

### 3.3 可用性需求

| 进度可见 | 长时间操作显示进度条和状态信息 |
| 配置灵活 | 支持自定义分类参数和AI模型配置 |

### 5.3 AI分类结果校验规则 ✅

| 校验项 | 规则 | 处理方式 |
|--------|------|----------|
| 完整性 | 每个书签必须有分类 | 缺失则记录警告,归入"未分类" |
| 数量匹配 | 分类数量 = 书签数量 | 不匹配则记录警告 |
| 路径有效性 | 分类路径必须在预定义列表中 | 无效路径则归入"未分类" |
| 幻觉检测 | 分类名称不能包含URL或特殊字符 | 检测到幻觉则重新请求或归入"未分类" |
| 理由完整性 | 每个分类应包含归类理由 | 缺失理由则记录警告 |

## 6. 约束与假设

### 6.1 约束条件

| 约束类型 | 描述 |
|----------|------|
| API限制 | 浏览器扩展API无法直接获取书签访问历史 |
| 成本限制 | AI模型调用存在延迟和费用成本 |
| 隐私保护 | 用户书签数据隐私保护要求 |
| 权限限制 | 仅能申请必要的浏览器权限 |
| 存储限制 | chrome.storage.local有容量限制 |

### 6.2 假设条件

| 假设类型 | 描述 |
|----------|------|
| 用户前提 | 用户已拥有AI模型API访问权限 |
| 用户意愿 | 用户愿意接受AI的分类建议 |
| 环境支持 | 浏览器支持Manifest V3扩展 |
| 网络条件 | 用户有稳定的网络连接访问AI API |
